## 3. Сопоставление НП-строка ↔ карточка LabResta/Horoshop

### Главный факт (определяет всю архитектуру)

Классическое article-anchor правило аудита (`sp.article == pp.display_article` И артикул встречается в `pp.name`/`pp.name_ru`) **здесь не работает**: у НП-эксклюзивных карточек в каталоге Horoshop `display_article` **пустой** (локальный снапшот: `has_display_article = 0` для всех Hurakan/Apach/Fagor/Tatra/Cold/Arris). Это согласуется с заметкой Cat-H walkthrough: «локальные feed'ы этих брендов не передают `<artikul>` в YML — правильного значения для display_article нет».

→ **Единственный надёжный мост НП-строка → карточка Horoshop — это уже существующий `ProductMatch` (status confirmed/manual)**, связывающий `SupplierProduct`(sup 2, `article` = НП col B) ↔ `PromProduct`. Ключ импорта в Horoshop = `PromProduct.external_id` (Horoshop «Артикул»/Identifier_tovaru). Цепочка:

```
НП-фид (col B Артикул) ── == ──► SupplierProduct.article (supplier_id=2)
                                        │  ProductMatch (confirmed|manual)
                                        ▼
                                  PromProduct.external_id  ──► карточка Horoshop
```

### Покрытие (локальный снапшот `instance/labresta.db`, read-only — НЕ прод)

| Метрика | Значение |
|---|---|
| НП SupplierProduct (sup 2, не удалённые) | **690** (= ровно число строк фида) |
| Из них с артикулом | 690 |
| Из них с confirmed/manual матчем | **370 (≈ 54 %)** |
| Без финального матча (нет карточки-цели) | **≈ 320 (≈ 46 %)** |

Каталожные PP по 9 брендам (locally): Hurakan 237 (matched 206), Apach 125 (89), Fagor 30 (13), Tatra 28 (22), Cold 3 (3), Arris 5 (5), Project Systems/Astoria/Maxima 0 в каталоге.

**Историческая справка** (`instance/NP-GAPS-REPORT.md`, 2026-04-22): тогда было 142 confirmed/manual; сейчас 370 — матчинг активно прогрессировал (chunk/audit-проходы, ручные подтверждения). Незакрытый хвост из того отчёта: A1 ≈ 100 пар уже в `candidate` ждут ручного подтверждения в `/matches`; A2 ≈ 105 пар матчер пропустил (баг pure-letter fast-path с цифрами в артикуле).

> ⚠ **Caveat:** локальная БД — dev-снапшот, может отставать/опережать прод. Авторитетное покрытие **обязано** быть пересчитано на проде (read-only) в момент исполнения. Цифра 370/690 — индикативная, для планирования объёма.

### Что это значит для объёма

- **Импортировать тело+фото можно только для матченных НП SKU** (≈370 на снапшоте) — у остальных ≈320 нет карточки-цели в Horoshop.
- Непокрытые ≈320 — заблокированы до улучшения матчинга (отдельная задача: добить A1 bulk-confirm + починить A2 fast-path). Не входят в первый прогон этой задачи.

## 4. Механизм доставки тела в Horoshop — две возможные дороги

LabResta не пишет в Horoshop напрямую; Horoshop **импортирует фид/файл**, который отдаёт LabResta (инвариант #13/#14).

**Текущее состояние генератора (`yml_generator._build_offer_xml`):** в `<offer>` пишутся `name`, `name_ru`, `price`, `<description>` (из `pp.description_ua`), `<description_ru>` (из `pp.description_ru`). **`<picture>` НЕ генерируется вообще.** Описание берётся из `PromProduct`, не из supplier feed.

| | Route A — через YML LabResta | Route B — прямой импорт-xlsx в Horoshop |
|---|---|---|
| Суть | НП desc/photo → `PromProduct.description_ua/ru/image_url` → пер-supplier/custom YML → Horoshop тянет | Сборка Horoshop-формат xlsx (ключ `external_id`, только колонки Опис ua/ru + Фото) → ручной импорт в админке Horoshop с профилем, обновляющим ТОЛЬКО эти поля |
| Нужен код в репо | **Да:** (1) отдельный НП-импортёр desc/photo (НЕ generic excel_parser; W1 на main), (2) запись в PromProduct, (3) добавить `<picture>` в `_build_offer_xml`, (4) перегенерация фида | **Нет** кода в репо; нужна настройка профиля импорта на стороне Horoshop |
| Whitelist полей | через состав `<offer>` (контролируем) | через профиль импорта Horoshop (точечно по колонкам) |
| Затирание прочих полей | зависит от Horoshop import-настроек | зависит от Horoshop import-настроек |
| Минус | правки кода в репо с активным W1; нет поддержки `<picture>` | требует доступ/знание профиля импорта Horoshop (серверная сторона — неизвестно) |

**Затирание других полей при частичном импорте регулируется настройками профиля импорта на стороне Horoshop, а не кодом LabResta.** Предупреждение про wipe в `catalog_import.save_catalog_products` относится к *входящей* БД LabResta, НЕ к Horoshop. → это открытый вопрос к Yana (см. QUESTIONS.md).

## 5. Коллизия с аудитом перевода (W1/W2) — НЕТ

`SKIP-НП` в коде аудита (`scripts/_apply022_b9.py` и т.п.): для SKU 9 НП-эксклюзивных брендов W1/W2 **forward-only пропускают** — RU не переписывается, ячейки `nazv_ru`/тело в chunk-фикс-xlsx **не трогаются**, «тело придёт из фида НП позже» (= эта задача).

- W1/W2 владеют полями **имени/названия** (`title_uk`/`title_ru` ↔ `name`/`name_ru`) и для SKIP-НП их вообще не меняют.
- Эта задача владеет **описанием + фото** и имён не трогает.
- Поля **непересекающиеся** → жёсткой коллизии нет. Рекомендация по порядку — в PLAN.md (фаза 0): импорт тел запускать **после** того как W1/W2 закроют свой проход по затрагиваемым chunk'ам, чтобы не гнать две live-import операции одновременно (инвариант #13: одно живое изменение за раз).

## Ревизия 2 (2026-05-18, после обсуждения с Yana)

### R2.1 Тело в фиде = ровно то, что на сайте (доказано на HKN-PICO12M)

Сверка карточки `ХЛІБОРІЗКА HURAKAN HKN-PICO12M` (скрин сайта, вкладка «Опис») с фидом
(`scratch_pico12m.txt`): содержимое **совпадает слово в слово, строка в строку**. Единственная
разница — фид хранит описание одной строкой с тегами `<br>`, а Horoshop рендерит `<br>` как
перенос. → лить тело **как есть, с `<br>`**, без конвертации. RU (col Q) — настоящий отдельный
русский («настольная хлеборезка», «Производительность», «нержавеющая сталь»). GATE подтверждён повторно.

### R2.2 Фид ДАЁТ галерею (правка §2 — было «1 фото/товар», НЕВЕРНО)

Колонка D — `;`-разделённый список URL. Распределение числа фото (`scratch_photo_dist.txt`):

| Фото на товар | Все 690 | 9 scope-брендов (547) |
|---|---|---|
| 1 | 489 | 383 |
| 2 | 46 | 34 |
| 3 | 62 | 56 |
| 4 | 46 | 38 |
| 5 | 21 | 19 |
| 6 | 13 | 9 |
| 7 | 5 | 4 |
| 8 | 5 | 2 |
| 9 | 2 | 1 |
| 10 | 1 | 1 |

→ **164 из 547 scope-товаров несут 2+ фото** (до 10). Решение Yana — лить **всю галерею**
(`image_url` = 1-е, `images` JSON = все). HKN-PICO12M в фиде имеет 2 фото — на сайте сейчас 1.

### R2.3 Формат тела по всему фиду (правка §1 — чище, чем показала выборка 60)

Полный проход 690 строк: **0 пустых описаний** (и uk, и ru есть у всех 690). Весь HTML —
**только `<br>`** (ни `<p>/<ul>/<li>/<div>/<table>`). Формат предсказуемый, парсинг тривиален.
(Доля идентичных ru==uk по полному набору отдельно не считалась; emptiness = 0 → Q7 почти снят.)

### R2.4 Код-путь Route A (грунтовка спеки, проверено по коду)

- `SupplierProduct` уже имеет `description`(Text), `image_url`(Str500, «Main picture from feed»),
  `images`(Text, «All picture URLs, JSON array»), `params`(Text JSON) — схема выдержит галерею.
  **Но `description` одно, без ua/ru** → нужно решение по хранению `description_ru` (QUESTIONS Q-стор).
- `excel_parser.parse_excel_products` по контракту возвращает только
  external_id/name/brand/model/article/price/currency/available — **desc/photo выбрасываются**.
  Прецедент обхода — `app/services/rp_parser.py` (отдельный парсер для РП Україна). → Route A
  делает `np_parser.py` по этому образцу, не трогая generic excel_parser.
- `yml_generator._build_offer_xml`: оффер строится из **`pp`** (карточка Horoshop), не из sp.
  `<description>`←`pp.description_ua`, `<description_ru>`←`pp.description_ru`,
  `id`/`vendorCode`←`pp.external_id`. **`<picture>` не эмитится ни в одном фиде.**
- `regenerate_supplier_feed(supplier_id)` существует → `labresta-feed-<slug>.yml`
  (НП slug=`novyy-proekt`), фильтр по supplier_id, только `status IN (confirmed,manual)`
  **и `published=1`**. Это и есть «фид по ссылке вместе с матчами» из решения D.
- Минимальный путь: (1) `np_parser` → desc/photo на sp; (2) идемпотентный `np_sync_bodies`
  пишет 4 поля sp→pp для матченных; (3) `<picture>` в `_build_offer_xml` (главное+галерея,
  под `if`); (4) Horoshop тянет тот же URL по расписанию = рекуррентность.

### R2.5 Корректировка §4 — Route выбран

§4 описывал A vs B как открытый выбор. **Зафиксировано: Route A**, рекуррентно. Route B
(ручной разовый xlsx) отклонён — противоречит требованию рекуррентности (решение E).

хліборізка.<br>Продуктивність: до 240 батонів/год.<br>...» (318 симв).
→ Импорт канарейки **ЗАМЕНИТ** прозу на спис-фида. Это РЕАЛЬНОЕ видимое
изменение, НЕ no-op. R2.1 («карточка уже = фид») по скрину Yana был про
другой источник/момент; live-факт сейчас — иной. Q-перезапись становится
АКТУАЛЬНЫМ решением (фид=правда → проза затрётся спис-листом).

**R4.2 RU-описание на сайте СЕЙЧАС ОТСУТСТВУЕТ (сильный аргумент ЗА задачу).**
На `/ru/` странице меню/хром переключается на русский, но **тело «Опис»
остаётся УКРАИНСКИМ** (идентично UA, 728 симв укр текста). Т.е. русского
описания у НП-карточек на живом магазине сейчас НЕТ. `<description_ru>` из
фида это и закрывает — канарейка добавит настоящий RU там, где его нет.

**R4.3 Канарейка = НЕ чистый no-op контроль (правка Фазы 4).** PLAN Ф4
ожидал «карточка уже = фид → чистый контроль». Факт: тест покажет (а) замену
тела проза→спис, (б) появление RU, (в) +2-е фото. Всё это — нужный эффект,
но Yana должна подтвердить, что замена прозы на фид-спис желаема (Q-перезапись).

## Ревизия 6 (2026-05-19, «ОБРОБНИК»-вопрос ЗАКРЫТ дефинитивно, doc-proven)

Перекрывает любые прежние допущения про сохранение/рекуррентность маппинга.

**R6.1 У Horoshop НЕТ функции «сохранить обробник/маппинг».** Доказано 6
источниками help.horoshop.ua (1684865, 6941142, 7155564, 6675136 Постачальник-
beta, 7156064 архив-2022, Skynum-integration) — единогласно: импорт товаров/
описаний/фото = **ручная разовая операция** (файл ИЛИ по URL — есть поле
«File URL»), **без** сохранения маппинга, **без** расписания, **без**
автоповтора. Это платформенное ограничение Horoshop.

**R6.2 Нативная схема (доказано live 2026-05-19) маппится сама.** Заголовки
`[КАТАЛОГ] <поле>` → Horoshop авто-маппит ВЕСЬ текст/цену/наличие/имя,
RU→RU UA→UA (кросс-порча исходного бага решена содержимым файла). Это даёт
рекуррентную устойчивость БЕЗ перемаппинга колонок — но НЕ убирает оператора.

**R6.3 Достижимый максимум рекуррентности.** Оператор каждый импорт: вставить
URL фида → вручную выставить **Галерея**(id39) на фото-колонку (Image/Gallery
Horoshop BY DESIGN никогда не авто-маппит) → «Імпортувати». ≈3 клика, ноль
работы с колонками. Для основного фида матчера (нет фото→нет Галереи): URL +
Імпортувати, 2 клика. Полностью без оператора xlsx в Horoshop невозможно.

**R6.4 Плановый авто-pull у Horoshop ТОЛЬКО для YML** (авто-маппинг по тегу,
нет UI маппинга) — а YML дропает `<name_ru>`/`<description_ru>` (= исходный
«в ру сел укр» баг, обробником НЕ лечится: YML-парсер Horoshop эти теги не
узнаёт). Это переводит выбор в плоскость: нативный-xlsx (верно, ручной
≈3-клик триггер) vs YML (авто, RU сломан).

**R6.5 Гипотеза «экран планового URL-импорта с сохранённым маппингом» —
ОПРОВЕРГНУТА.** Такого экрана в Horoshop нет. 85 «обробників» в select#4
превью = пер-категорийные хендлеры характеристик (Довідники), не шаблоны
импорта. Полный разбор + рекомендации (R1 принять ручной триггер / R2 копать
YML-import-профиль) — CANARY-IMPORT-GUIDE.md §6.
