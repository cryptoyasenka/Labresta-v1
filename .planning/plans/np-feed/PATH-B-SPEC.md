## 0. Что путь B делает (одна фраза)

Главный матчер-фид остаётся YML по ссылке с авто-расписанием (оператор не
нужен ВООБЩЕ), но из `<offer>` убираются `<name>`/`<name_ru>`/
`<description>`/`<description_ru>` → Horoshop-YML-парсеру нечего крест-
маппить → накопительная порча «укр в ру» физически невозможна. Цена / старая
цена / наличие / валюта / артикул синкаются авто как раньше.

## 1. B3 — собранные ФАКТЫ (read-only, НЕ решения)

### 1.1 Единственная точка эмиссии текстовых полей
`_build_offer_xml` (`yml_generator.py:80-118`) — ЕДИНСТВЕННАЯ функция,
эмитящая полную форму оффера с именем/описанием. Точные строки:

| Строка | Тег | Источник | Куда садится в Horoshop | Путь B |
|---|---|---|---|---|
| `:95` | `<name>` | `match.feed_name or pp.name` (UA) | YML-парсер жёстко → «Назва модиф.(RU)» = **корень порчи** | **УБРАТЬ** |
| `:96-97` | `<name_ru>` | `pp.name_ru` (если непусто) | парсер не знает тег → дроп | **УБРАТЬ** |
| `:112-114` | `<description>` | `pp.description_ua` (CDATA) | → «Опис тов.(RU)» = порча | **УБРАТЬ** |
| `:115-117` | `<description_ru>` | `pp.description_ru` (CDATA) | дроп | **УБРАТЬ** |

### 1.2 Что в `_build_offer_xml` ОСТАЁТСЯ (НЕ трогать — это работа фида)
- `:92` `offer id=pp.external_id` — ключ update-by-article (Horoshop «Артикул»).
- `:93` `available=` — наличие.
- `:99` `<url>` = `pp.page_url` (если есть) — НЕ UA/RU-вектор; см. §4 откр.вопрос.
- `:100` `<price>` — главная цель фида.
- `:104-105` `<oldprice>` = supplier retail — анти-двойная-скидка (карточная
  «Знижка %» игнорируется когда есть oldprice). Снять = сломать ценообразование.
- `:107` `<currencyId>`.
- `:108` `<vendorCode>` = `pp.external_id`.
- `:109-110` `<vendor>` = `pp.brand` — текст, НО Horoshop-авто-маппинг
  `<vendor>`→«Бренд» это НЕ UA/RU-крест-порча. Strip имён/описаний достаточен
  для языкового бага; `<vendor>` ортогонален → см. §4 откр.вопрос.

### 1.3 Общий вызов — путь B бьёт ВСЕ ТРИ полных фида (инвариант #14)
`_build_offer_xml` зовётся ТОЛЬКО из `_generate_feed` (`:142`), общего
билдера. `_generate_feed` используют:

- `regenerate_yml_feed()` (`:256-265`) — **main**, `update_in_feed=True`.
- `regenerate_supplier_feed(supplier_id)` (`:268-282`) — **per-supplier**,
  `in_feed` НЕ трогает.
- `regenerate_custom_feed(match_ids)` (`:285-316`) — **custom**, `in_feed`
  НЕ трогает.

Докстринг модуля (`:11-13`) подтверждает прямым текстом: «Per-supplier and
custom feeds use the SAME offer shape as the main feed (name, vendorCode,
price, available, description)». → Правка `_build_offer_xml` без скоупа
автоматически убирает имена/описания из ВСЕХ ТРЁХ. Это и есть «blast radius»
из §8 B3 — теперь подтверждён точно (не «вероятно общий», а доказанно один
код-путь).

### 1.4 Узкие фиды уже «path-B-формы» — НЕ затрагиваются
`sync_prices` (`:339-401`) и `sync_availability` (`:404-459`) строят оффер
ИНЛАЙН, НЕ через `_build_offer_xml`, и уже БЕЗ name/description:
- prices-оффер: `id`+`vendorCode`+`price`+`currencyId`.
- availability-оффер: `id`+`vendorCode`+`available`.
Тесты это фиксируют (`test_yml_generator.py:388-391` и `:502-505`:
`assert offer.find("name") is None`). Подтверждает §8 B4 — НП-узкий by
design no-name. Путь B делает главный фид консистентным с узкими.

### 1.5 КРИТИЧНО: в YML два разных `<name>` — путать НЕЛЬЗЯ
- **shop-level** `_shop_skeleton():180` `etree.SubElement(shop,"name")="LabResta"`
  — метаданные магазина. Тест `test_export_yml_subset.py:107`
  `assert "<name>LabResta</name>" in text` это фиксирует.
- **offer-level** `_build_offer_xml:95` `<offer><name>` — товарное имя,
  ИМЕННО его крест-маппит Horoshop-парсер.

**Путь B трогает ТОЛЬКО offer-level. Shop `<name>` остаётся** — иначе
YML невалиден и `test_export_yml_subset.py:107` краснеет зря.

## 3. Скоуп B3 — ОПЦИИ (решение Yana / код-фазы, НЕ здесь)

`_build_offer_xml` общий → надо выбрать.

**Опция 1 — strip у всех трёх фидов (просто, консистентно).**
Удалить блоки A+B прямо в `_build_offer_xml`. Main/per-supplier/custom
теряют имя/описание разом. Аргумент «за»: Horoshop владеет именами
(B1 доказал — фид только update-by-article существующих карточек, ничего
не обезличивается); консистентно с узкими фидами (§1.4). Минус: `feed_name`-
фича умирает для фида везде (§1.7); per-supplier/custom тоже больше не
несут описаний (если кто-то на это полагался — проверить потребителей).

**Опция 2 — флаг скоупа, strip только main.**
`_build_offer_xml(parent_el, match, *, emit_text: bool = True)`;
`_generate_feed` пробрасывает `emit_text`; `regenerate_yml_feed` зовёт с
`emit_text=False`, per-supplier/custom — дефолт `True`. Плюс: per-supplier/
custom и `feed_name` сохраняют поведение; чинит ровно вектор порчи (main =
тот, что Horoshop тянет по расписанию). Минус: больше кода, разъезд формы
между фидами (мягко против духа инварианта #14 — «единый формат»), узкие
фиды и так разъезжаются, так что прецедент есть.

**Факт для решения:** порчу по расписанию накапливает ТОЛЬКО плановый
авто-pull, а это main YML по ссылке. Per-supplier/custom Horoshop по
расписанию НЕ тянет (доказано Ревизия 6: плановый авто-pull = только один
YML-URL). → Языковой баг закрывается даже опцией 2 (узкий скоуп достаточен
для собственно бага). Опция 1 шире (консистентность/чистота), опция 2 —
минимально-инвазивна. Выбор — Yana/код-фаза.
