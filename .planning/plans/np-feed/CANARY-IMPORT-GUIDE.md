## 3b. ПОЧЕМУ Horoshop не взял канарейку — ROOT CAUSE (из кода, read-only)

Сверил ручную канарейку с рабочим генератором (`yml_generator._shop_skeleton`,
`_write_xml_atomic`, `_build_offer_xml`) + `scripts/export_yml_subset.py`.
Рабочий фид, который Horoshop УЖЕ принимает, пишется так:

- **DOCTYPE** `<!DOCTYPE yml_catalog SYSTEM "shops.dtd">` — у моей канарейки ЕГО НЕТ.
- `date` = реальный таймстемп `"%Y-%m-%d %H:%M"` — у канарейки `date="canary-test"`
  (НЕВАЛИДНО → парсер Horoshop может отбросить весь каталог).
- Оффер рабочего фида: `id`+`available`+`<name>`+`<name_ru>?`+`<url>?`+`<price>`+
  `<oldprice>?`+`<currencyId>`+`<vendorCode>`+`<vendor>?`+`<description>`CDATA+
  `<description_ru>`CDATA. **`<picture>` НЕ эмитит НИ ОДИН существующий фид.**
- Канарейка: без DOCTYPE, без name/price/currencyId, +`<picture>` (нестандарт).

→ Скорее всего Horoshop молча отбраковал файл из-за **отсутствия DOCTYPE и/или
невалидного `date`** (структура ≠ та, под которую настроен профиль импорта).
Это НЕ «нужен деплой» по сути, НО интуиция Yana частично верна (см. ниже).

### Развилка (РЕШЕНИЕ Yana — не угадываю)
- **(A)** Пересобрать канарейку ТОЧНО машинерией рабочего фида: DOCTYPE +
  валидный `date` + полный shop-скелет + стандартная форма оффера, добавив
  `<description>`/`<description_ru>` = тело НП и `<picture>`. Риск: `<picture>`
  не эмитил ни один фид → ни один Horoshop-профиль его не «видел» (непротестировано).
  Плюс конфликт с no-name дизайном: если профиль требует `<name>` для
  обработки оффера — придётся добавить имя (домен W1).
- **(B)** Признать: достоверный тест тела+фото требует Phase 3 pipeline
  (enrich sp→pp в БД) + деплой → ровно интуиция Yana «только с деплоя».
  Тогда канарейка-через-тоннель отменяется, тест переносится за сигнал 0.5.

`export_yml_subset.py` не выручает: он режет 1 оффер из УЖЕ собранного
`instance/feeds/labresta-feed.yml`, где `<description>`=текущий `pp.description_ua`
(НЕ тело НП) и нет `<picture>` — это протестирует доставку, но не НП-контент.

## 6. «ОБРОБНИК» / рекуррентность — ДЕФИНИТИВНЫЙ ОТВЕТ (2026-05-19, doc-proven)

**Вопрос Yana:** «сохрани обробник… чтобы без оператора каждый раз».
**Ответ: в Horoshop ТАКОЙ функции НЕТ.** Доказано 6 авторитетными источниками
help.horoshop.ua: 1684865 (import from file), 6941142 (adding via import),
7155564 (updating product data), 6675136 (Постачальник beta), 7156064 (2022
архив), Skynum integration — **все сходятся**:

| Что есть в Horoshop | Что НЕТ |
|---|---|
| Импорт из файла **или по URL** (есть поле «File URL» — ссылку можно вставить) | Сохранения маппинга/шаблона/обробника для переиспользования |
| Авто-маппинг колонок с заголовком `[КАТАЛОГ] <поле>` (доказано live) | Расписания / автоповтора xlsx-импорта по URL |
| «Постачальник» (Товари→Довідники→Постачальники): выборочный апдейт только товаров этого поставщика | Персистентного пер-поставщик маппинга (маппинг всё равно вручную каждый раз) |
| Плановый авто-pull по URL — **только YML** (авто-маппинг по тегу) | — а YML дропает `<name_ru>`/`<description_ru>` (исходный баг, обробником НЕ чинится) |

**Гипотеза прошлого снапшота** («рекуррентный маппинг живёт в конфиге планового
URL-импорта, отдельный экран») — **ОПРОВЕРГНУТА**. Такого экрана в Horoshop
нет. 85 «обробників» в select#4 превью = пер-категорийные хендлеры
характеристик (Довідники / attribute value lists), НЕ шаблоны импорт-маппинга.

### Что это значит операционно (честно)

**Полностью без оператора xlsx-канарейку в Horoshop сделать НЕЛЬЗЯ** — это
ограничение платформы, не моё. Достижимый максимум (и он хороший):

> Нативные `[КАТАЛОГ] <поле>` заголовки → Horoshop **сам авто-маппит** ВЕСЬ
> текст/цену/наличие/имя UA-в-UA RU-в-RU (БЕЗ ручного перемаппинга колонок,
> кросс-порча RU решена на уровне содержимого файла). Оператору каждый импорт:
> (1) вставить URL фида, (2) вручную выставить **Галерея** (id39) на фото-колонку
> [Image/Gallery Horoshop BY DESIGN никогда не авто-маппит], (3) «Імпортувати».
> ≈3 клика, ноль работы с колонками, повторяемо и устойчиво.

Для **основного фида матчера** в нативной xlsx-схеме Галереи нет (фото нет) →
авто-маппится ВСЁ → оператору: вставить URL + «Імпортувати» (2 клика), ноль
работы с колонками. Тоже ручной триггер (планового xlsx в Horoshop нет), но
без перемаппинга. Полностью авто-YML остаётся сломан по RU.

### Рекомендация

Канарейка-механизм ТЕХНИЧЕСКИ ЗАКРЫТ и доказан live: нативная xlsx-схема
маппится верно, RU чинится. «Обробник» сохранять **негде и нечего** — это

## 8. ПРЕВЬЮ ЭФЕМЕРНО ПОД УДАЛЁНКОЙ — ИМПОРТ ДЕЛАЕТ Yana ОДНИМ ЗАХОДОМ (2026-05-19)

**Находка (пост-компакт, по «все, тестируем импорт?»):** превью, оставленное
«открытым на руку Yana», между сессиями НЕ переживает. Модалка «Імпорт»
закрывается, legacy-iframe (`data.php?handler=17`) дрейфует на page-editor
(`edit.php?handler=4`, id=92 затем id=927). Пере-стейдж с нуля прошёл и
**в 3-й раз подтвердил механизм**: канарейка залилась, ПРЕВЬЮ отрисовалось,
**11/12 полей авто-сели** (Артикул, Цена, Наличие, Отображать, Назв.модиф
UA/RU, Опис UA/RU, Бренд, Старая цена, Валюта), фото-колонка опознана **по
данным** (col idx 4, значение `https://np.com.ua/.../2-15-1.jpg;…`, маппинг
`(none)`=«Не імпортувати»).

**Почему удалённый пре-маппинг Галереи мной НЕнадёжен:**
- `vue-treeselect.forest.nodeMap` пуст до `.openMenu()`; `.openMenu()` и любой
  re-render **пере-индексируют** `[class*=treeselect]` → ссылка на виджет
  теряется между CDP-round-trip'ами (каждый eval ≈ секунды).
- **Корневая причина дрейфа iframe = мой форсированный
  `iframe.contentWindow.location.href=…` воюет с родительским SPA.** Если
  iframe НЕ трогать — SPA сам корректно держит его на `handler=17`
  (проверено: tab `/edit/products/all`, iframe0 `data.php?handler=17`, чисто).

**Решение по исполнению (совпадает с инвариантом #13):** удалённый
пре-маппинг отменён как ненадёжный. Импорт-канарейку Yana выполняет **сама,
одной непрерывной человеческой сессией** в живом табе (человек держит
модалку в фокусе, проходит upload→preview→1 клик Галерея→Імпортувати за
секунды — это штатный merchant-UX Horoshop, без CDP-пауз для дрейфа).
Моя роль: (1) механизм зелёный (доказан 3×); (2) точный путь (этот §
+ §7 + FINAL-MODEL §3); (3) канарейка на диске
(`_serve\canary-HKN-PICO12M.xlsx`, 5939 b); (4) **post-import read-only
верификация live-карточки** (UA/RU/фото/цена) — по сигналу Yana.

**Безопасность подтверждена:** «Імпортувати» НЕ жат; `DOM.setFileInputFiles`
= только клиентский парс-превью, в каталог НЕ пишет; репо/прод/туннель не
тронуты. Таб оставлен чистым и готовым к ручному импорту.
Техническая заметка для воспроизведения: путь к файлу в
`DOM.setFileInputFiles` — **forward-slash** (`C:/Users/...`), backslash
ломает JSON-парс параметров CDP.
