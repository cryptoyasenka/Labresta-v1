# MATCHER-FEED-FIX-SPEC — почему «в ру сел укр» и как чинить основной фид

**Статус:** диагноз подтверждён live (profile 1, 2026-05-19). Код в репо НЕ
менять, пока W1 на main (decision 0.5). Это спека — применит Yana/отдельный
заход, когда main свободен. Импорт/«Зберегти» профиля — рука Yana (#13).

## 1. Симптом
В Horoshop во многих карточках русское поле названия содержит украинский
текст. Появилось «само», накопительно — не разовое событие.

## 2. Корневая причина (тот же баг, что у канарейки)
Horoshop import авто-маппит колонку фида в поле каталога ТОЛЬКО когда имя
колонки совпадает с именем поля каталога (док help.horoshop + доказано живьём).

`app/services/yml_generator.py::_build_offer_xml` (read-only осмотр, стр. ~95):
- `<name>` = `match.feed_name or pp.name` → **украинское**
- `<name_ru>` = `pp.name_ru` → чистый русский
- `<description>` = `pp.description_ua`, `<description_ru>` = `pp.description_ru`
- `<picture>` в основном фиде **вообще не генерится**

Horoshop YML-парсер жёстко зовёт тег `<name>` колонкой
«Название модификации(RU)» → авто-кладёт **украинский** `<name>` в **русское**
поле. Тег `<name_ru>` он НЕ узнаёт → «Не імпортувати» (правильный русский
выбрасывается). То же с `<description>`(укр)→«Описание товара(RU)»,
`<description_ru>`→дроп. Каждая плановая синхра дописывает порчу → «накопительно
во многих карточках».

## 3. Наши данные ЧИСТЫЕ (важно)
Probe live `instance/labresta.db?mode=ro`: `prom_products` 5683, непустых
`name_ru` 5633, из них с украинской буквой і/ї/є/ґ — **ровно 1**. Порча НЕ в
БД/матчере, а в Horoshop-импорте. → Правильно сконфигурированный ре-импорт
**ПОЧИНИТ** русские названия (правильный `pp.name_ru` у нас есть).

## 4. Варианты фикса
- **Реком.: основной фид в НАТИВНОЙ Horoshop-схеме** (как канарейка / xlsx
  dealer-export): отдельные колонки, заголовок = ТОЧНОЕ квалифицированное имя
  поля каталога Horoshop. **Эмпирически подтверждено live (profile1,
  2026-05-19):** авто-маппер ловит колонку ⟺ заголовок == `[КАТАЛОГ] <leaf>`
  (top-level поля — голым именем, напр. `Артикул`). Точные строки, снятые из
  vue-treeselect forest (все под группой «КАТАЛОГ», id):
  `[КАТАЛОГ] Название модификации (UA)` id4 · `[КАТАЛОГ] Название модификации (RU)`
  id5 · `[КАТАЛОГ] Описание товара (UA)` id37 · `[КАТАЛОГ] Описание товара (RU)`
  id38 · `[КАТАЛОГ] Цена` id6 · `[КАТАЛОГ] Старая цена` id10 · `[КАТАЛОГ] Валюта`
  id19 · `[КАТАЛОГ] Наличие` id11 · `[КАТАЛОГ] Отображать` id12 · `[КАТАЛОГ] Бренд`
  id33 · `[КАТАЛОГ] Галерея` id39 (мультифото url;url) / `[КАТАЛОГ] Фото` id14
  (одиночное) · `Артикул` id0 (top-level, без префикса).
  С такими заголовками RU/UA имена и описания садятся в РАЗНЫЕ правильные поля
  (UA→UA, RU→RU) — кросс-мапа `<name>`(укр)→RU больше НЕ происходит. Авто-маппинг
  верный, без оператора, рекуррентно-устойчиво. Это правка доставки фида (новый
  генератор/формат), а не матчера; матчер-данные ок.
- **ВАЖНО для основного фида:** YML-теги `<name>`/`<name_ru>`/`<description*>`
  Horoshop НЕ узнаёт под этими именами (YML-парсер жёстко зовёт `<name>`
  колонкой «Название модификации(RU)» → та самая кросс-порча). Нативная схема
  требует СМЕНЫ ФОРМАТА выгрузки YML→xlsx/csv с заголовками выше, а не просто
  переименования тегов внутри YML.
- **Фото — в том же фиде, но маппинг one-time.** Подтверждено live (2-й
  preview 2026-05-19) + доком Horoshop (help.horoshop.ua/en/articles/8129124):
  фото-по-URL = ТОТ ЖЕ товарный импорт, колонка «Image»/«Gallery», URL через
  `;`. Все `[КАТАЛОГ] *` текст/цена авто, НО Image/Gallery Horoshop BY DESIGN
  никогда не авто-маппит («except for Image and Gallery — mapped manually»).
  Утилита «Імпорт зображень» = только локальные файлы по артикулу, под URL не
  годится. → Рекуррентно-безопасно: нативные заголовки (текст авто) + Галерею
  смапить ОДИН РАЗ и СОХРАНИТЬ обробник → дальнейшие синхры применяют сами.
  Для основного фида матчера это значит: при переводе на нативный xlsx можно
  добавить колонку `[КАТАЛОГ] Галерея` (если решим лить фото оттуда) — но текст
  RU/UA чинится и без неё; фото опционально, через тот же one-time обробник.
- Запас: оставить YML, один раз настроить и СОХРАНИТЬ корректный import-профиль
  Horoshop для основного фида. РИСК: Horoshop YML-парсер не отдаёт `<name_ru>`
  как маппируемую колонку (он её дропает) — тогда профилем русское имя не
  спасти. Перед выбором этого пути проверить на preview, какие именно source-
  колонки Horoshop вообще показывает для нашего YML (по факту: name_ru/
  description_ru не показываются как доступные цели — поэтому путь хрупкий).

> **ОБНОВЛЕНИЕ 2026-05-19 (doc-proven, 6 источников help.horoshop.ua).**
> «СОХРАНИТЬ import-профиль/обробник» в Horoshop **НЕВОЗМОЖНО** — такой фичи
> нет: импорт = ручная разовая операция (файл ИЛИ по URL), без сохранения
> маппинга / расписания / автоповтора. Плановый авто-pull по URL у Horoshop
> ТОЛЬКО для YML с авто-маппингом по тегу (а он и дропает name_ru). Значит:
> (1) нативный-xlsx путь = ручной триггер каждый раз (но БЕЗ перемаппинга:
> `[КАТАЛОГ] <поле>` авто-маппится, доказано live; осн.фид без фото = 2 клика
> URL+Імпортувати); (2) «запас с сохранённым YML-профилем» как способ
> рекуррентности **отпадает** — сохранять профиль негде. Остаётся открытым
> лишь: можно ли в YML-профиле Horoshop (если он есть в её плане/кабинете)
> вручную доназначить `<name_ru>`→RU-поле — но доки говорят, что парсер тег
> дропает ДО маппинга, т.е. путь почти наверняка мёртв. Полный разбор —
> `CANARY-IMPORT-GUIDE.md §6`, `FINDINGS.md Ревизия 6`.

## 5. Где менять (когда main свободен)
- `app/services/yml_generator.py`: `_build_offer_xml` (~80-118), `_shop_skeleton`,
  `_write_xml_atomic` — формат выгрузки основного фида.
- `app/views/feed.py` — роут отдачи (Content-Type/имя файла при смене YML→xlsx/csv).
- Узкие/per-supplier/custom фиды (инвариант #14) — единый формат, не разъехаться.
- НЕ трогать матчер/`pricing.py`/гейты — данные корректны, проблема в доставке.

## 6. Протокол безопасной выкатки (live-store, #13)
1. Сначала 1 товар (как канарейка) в новом формате → Yana проверяет preview:
   укр→UA-поле, рус→RU-поле, без «Не імпортувати». Импорт жмёт Yana.
2. Убедиться, что корректный импорт ПЕРЕЗАПИШЕТ испорченные RU правильным
   `pp.name_ru` (recovery), не наоборот.
3. Согласовать порядок с W1/W2 (они правят имена/переводы тех же карточек) —
   одно живое изменение за раз, не столкнуться.
4. Только потом масса.

## 7. Связь с НП-канарейкой
Один корень → одно решение (нативная схема). Канарейку добиваем первой как
наименее рисковый полигон; на ней же подтверждаем нативные заголовки, которые
Horoshop авто-узнаёт, и переносим в основной фид.

## 8. РЕШЕНИЕ Yana 2026-05-19 — путь B (YML без имён/описаний)

После доказательства (доки+кабинет), что сохранённого import-обробника/
маппинга в Horoshop НЕТ и YML-парсер захардкожен, Yana выбрала разделение:

- **НП (канарейка + товары НП)** → ручной импорт **Excel-файлом** в нативной
  `[КАТАЛОГ]`-схеме: авто-маппинг, оператор колонки не трогает, ≈3 клика
  (вставить → Галерея вручную → Імпортувати). Settled, работает.
- **Общий матчер-фид** → остаётся **YML по ссылке с авто-расписанием**
  (оператор не нужен), НО из `_build_offer_xml` **убираются**
  `<name>`/`<name_ru>`/`<description>`/`<description_ru>`. Фид перестаёт
  трогать имя/описание → кросс-порча «укр в ру» физически невозможна (нечего
  крест-маппить). Цена/старая цена/наличие/валюта/артикул синкаются авто.

### Следствия варианта B — Yana принимает ОСОЗНАННО (не «turnkey»)

- **B1 (новые товары) — РАЗРЕШЕНО БЛАГОПРИЯТНО (read-only код, 2026-05-19).**
  Доказательство из репо: `app/services/catalog_import.py:4-5,237` — таблица
  `PromProduct` это снимок **ПОЛНОГО Horoshop catalog export** (upsert по
  `external_id` = Horoshop «Артикул»). `app/services/yml_generator.py:44-56`
  (`_query_published_matches`: confirmed/manual + `published`) и `:80-108`
  (`_build_offer_xml`: offer `id`/`vendorCode` = `pp.external_id`) — главный
  фид строит офферы ТОЛЬКО из матчей на существующие pp. Значит каждый товар
  фида = карточка, **уже существующая в Horoshop и уже имеющая имя** (pp
  пришёл из самого Horoshop-экспорта). Фид по построению содержит только
  артикулы, которые Horoshop уже знает → импорт = update-by-article, НЕ
  create. Удаление `<name>` (путь B) НИЧЕГО не обезличивает: имена живут в
  карточках Horoshop независимо от фида; новые товары заводятся отдельным
  каталожным процессом Horoshop (имя ставится там при создании). B1 закрыт,
  отдельный канал имени для «новых из фида» НЕ нужен.
- **B2 (B чинит только БУДУЩЕЕ, не лечит ПРОШЛОЕ).** B останавливает
  накопление порчи, но НЕ исправляет уже испорченные RU-имена в Horoshop
  (фид их больше не трогает вообще → и не «recovery»). Лечение прошлого =
  аудит перевода **W1/W2** (их chunk-проход именно это и делает) ЛИБО разовый
  корректирующий нативный-xlsx импорт правильных имён. B + аудит W1/W2 вместе
  = полное решение (B = «остановить кровь», W1/W2 = «залечить»). §6 п.2
  («импорт ПЕРЕЗАПИШЕТ испорченные RU») при пути B **не применяется** —
  recovery идёт не через фид.
- **B3 (blast radius).** `_build_offer_xml` — общий для main/per-supplier/
  custom фидов (инвариант #14, NP-план R3.1 требовал его НЕ трогать).
  Удаление name/description затронет ВСЕ фиды на нём. При кодовой фазе
  решить: убрать имена из всех (консистентно: Horoshop владеет именами) или
  скоупить флагом только для main. Дизайн-вопрос фазы, не сейчас.
- **B4 (узкие фиды).** НП-узкий фид и так был «no-name» by design — путь B
  делает главный фид консистентным с ним. Тела НП-брендов приходят НП-файлом
  (отдельный канал), не из YML → конфликта нет.

Кодовая правка — `_build_offer_xml` + проверка B1/B3 — **только после
освобождения main от W1 (сигнал 0.5)**. Сейчас: только спека, кода в репо нет.

> **РАЗВЁРНУТО → `PATH-B-SPEC.md` (2026-05-19).** Точные правки

### Уточнение Yana 2026-05-19 (2-я итерация — финальная модель)

Два канала, без вариантов (детали `PATH-B-SPEC.md §0.1`):
- **Канал 1 — все матчер-фиды по ссылке (общий + по поставщику + кастомные):**
  обновляют **ТОЛЬКО цену и наличие** (несущие: id/vendorCode/currencyId/
  price/oldprice/available). Имя, `name_ru`, описание, `description_ru` И
  **бренд/`<vendor>`** — все убираются. Оператор не нужен, расписание.
- **Канал 2 — НП-контент (описание + бренд + производитель + фото),**
  только эксклюзивные бренды «Новый проект», РЕДКО, вручную Excel-файлом в
  нативной `[КАТАЛОГ]`-схеме (канареечная ветка).
Это решает §8 B3 (скоуп = все матчер-фиды, не флаг-main) и снимает
открытый вопрос по `<vendor>` (бренд → Канал 2, не в матчер-фиде).
**РЕШЕНО Yana 2026-05-19: ДА** — названия UA/RU включаются в Excel
Канала 2. Состав Канала 2 закрыт: название UA/RU + описание + бренд +
производитель + фото. Источник правильных имён = W3 (W1/W2 НП ПРОПУСКАЮТ).
Остаток по матчер-фиду: только сигнал 0.5 (W1 освободит main).
