## Флаг 1 (b4, SKU 26-32) — Tefcold ≠ НП «COLD»? (классификация SKIP-НП)

**Контекст.** Зонд-скрипт делает наивную substring-проверку бренда
против НП-эксклюзивного списка. Для бренда **Tefcold** substring
`'cold' ⊂ 'tefcold'` → зонд печатает `NP-brand? = True ['cold']` для
SKU 26, 27, 28, 29, 30, 31, 32 (7 SKU).

**Моя трактовка (применена в b4).** Правило SKIP-НП сформулировано как
**членство в бренд-SET** {HURAKAN/Hurakan/Хуракан, APACH/Apach/Апач,
FAGOR/Fagor/Фагор, TATRA/Tatra/Татра, **COLD/Cold/Колд**, PROJECT
SYSTEMS, ASTORIA/Astoria, ARRIS/Arris, MAXIMA/Maxima}, а **не**
substring. **Tefcold** — это самостоятельный датский производитель
(Tefcold A/S, Дания), это **не** НП-эксклюзивный бренд «COLD/Колд».
Word-boundary проверка (`(?<![A-Za-z…])COLD(?![A-Za-z…])`) даёт
NP-hit `[]` для всех 8 SKU 25-32 (скрипт `_apply026_b4.py` это
эмпирически доказал: naive substring COLD-хитов = 7, word-boundary = 0).

**Решение в b4.** SKU 25 (Tehma) + SKU 26-32 (Tefcold) обработаны как
**обычные бренды → blknochg** (genuine отдельный корректный RU перевод
поставщика, `desc UA==RU` False, `nm_ru`==`nazv_ru` чистый рус.). Это
**не** SKIP-НП. Практический эффект для xlsx идентичен в любом случае:
blknochg тоже **не трогает ячейки** chunk-026-fixed (LIVE-магазин,
genuine RU не переписываем). То есть даже при противоположном решении
содержимое выгрузки то же — расхождение только в категории отчёта
(blknochg vs SKIP-НП), не в данных.

**Что нужно от тебя (необязательно — это override-флаг).**
Подтверди трактовку «Tefcold = обычный бренд, НЕ НП-эксклюзив COLD»
(ожидаемо — так и есть), **или** если по бизнес-логике Tefcold-позиции
тоже идут из фида НП и должны числиться как SKIP-НП — скажи, и я
переклассифицирую будущие батчи (ретро-прохода по уже закрытым нет;
forward-only). По умолчанию продолжаю трактовать Tefcold как обычный
бренд (blknochg).
