## Флаг 1 (b4, SKU 26-32) — Tefcold ≠ НП «COLD»? (классификация SKIP-НП)

**Контекст.** Зонд-скрипт делает наивную substring-проверку бренда
против НП-эксклюзивного списка. Для бренда **Tefcold** substring
`'cold' ⊂ 'tefcold'` → зонд печатает `NP-brand? = True ['cold']` для
SKU 26, 27, 28, 29, 30, 31, 32 (7 SKU).

**Моя трактовка (применена в b4).** Правило SKIP-НП сформулировано как
**членство в бренд-SET** {HURAKAN/Hurakan/Хуракан, APACH/Apach/Апач,
FAGOR/Fagor/Фагор, TATRA/Tatra/Татра, **COLD/Cold/Колд**, PROJECT
SYSTEMS, ASTORIA/Astoria, ARRIS/Arris, MAXIMA/Maxima}, а **не**
substring. **Tefcold** — это самостоятельный датский производитель
(Tefcold A/S, Дания), это **не** НП-эксклюзивный бренд «COLD/Колд».
Word-boundary проверка (`(?<![A-Za-z…])COLD(?![A-Za-z…])`) даёт
NP-hit `[]` для всех 8 SKU 25-32 (скрипт `_apply026_b4.py` это
эмпирически доказал: naive substring COLD-хитов = 7, word-boundary = 0).

**Решение в b4.** SKU 25 (Tehma) + SKU 26-32 (Tefcold) обработаны как
**обычные бренды → blknochg** (genuine отдельный корректный RU перевод
поставщика, `desc UA==RU` False, `nm_ru`==`nazv_ru` чистый рус.). Это
**не** SKIP-НП. Практический эффект для xlsx идентичен в любом случае:
blknochg тоже **не трогает ячейки** chunk-026-fixed (LIVE-магазин,
genuine RU не переписываем). То есть даже при противоположном решении
содержимое выгрузки то же — расхождение только в категории отчёта
(blknochg vs SKIP-НП), не в данных.

**Что нужно от тебя (необязательно — это override-флаг).**
Подтверди трактовку «Tefcold = обычный бренд, НЕ НП-эксклюзив COLD»
(ожидаемо — так и есть), **или** если по бизнес-логике Tefcold-позиции
тоже идут из фида НП и должны числиться как SKIP-НП — скажи, и я
переклассифицирую будущие батчи (ретро-прохода по уже закрытым нет;
forward-only). По умолчанию продолжаю трактовать Tefcold как обычный
бренд (blknochg).

**NON-FLIP.** Это classification doubt, не рассинхрон названия/тела →
`Открытых вопросов chunk-026 = 1` (OQ#1 SKU 15 из b2) НЕ меняется,
нового нумерованного OQ не вводится.

## Флаг 2 (b7, SKU 56) — GoodFood ICM15: источниковое расхождение габаритов UA↔genuine-RU (blknochg)

**Контекст (на момент записи: 56/70 — батч b7 SKU 49-56).** SKU 56
`759801055` GoodFood ICM15 — **blknochg** (`desc UA==RU` **False**,
`nm_ru`==`nazv_ru` чистый рус., genuine отдельный корректный RU
перевод поставщика, LIVE-магазин Horoshop → ячейки chunk-026-fixed
НЕ трогаем). В одном `<li>` габаритов источник рассинхронен между
языками:

- UA-тело: `Габаритні розміри 240 х 390 х 36 мм.` (разделитель —
  Cyrillic `х` U+0445; третье число `36`)
- genuine RU-тело: `Габаритные размеры 240*390*326 мм.` (разделитель
  `*`; третье число `326`)

Остальные числа совпадают (240, 390; масса 10,56 кг; 140 Вт; R600a;
1,5 литра). UA-сторона, судя по ряду, потеряла цифру: `36` против
`326` (вероятно UA-опечатка — пропущена `3` в `326`).

**Моя трактовка (применена в b7).** Это **blknochg** — genuine RU
LIVE-магазин, перезаписывать ячейки нельзя без явного go-ahead Yana +
safe mode. Расхождение в **UA-источнике** (UA-тело в RU-выгрузку не
идёт), не в customer-facing RU. RU-значение `240*390*326` выглядит
корректным (полные 3 цифры). Ячейки chunk-026-fixed НЕ трогаем,
genuine RU остаётся как есть.

**Что нужно от тебя (необязательно — это override-флаг).** Если по
бизнес-логике нужно поправить UA-тело карточки Horoshop `36`→`326` —
скажи; это правка на стороне магазина, вне scope перевод-аудита
(forward-only, ретро-прохода нет). По умолчанию ничего не трогаю.

**NON-FLIP.** Источниковое расхождение в UA-теле blknochg-позиции, не
customer-facing UA↔genuine-RU рассинхрон названия и не body↔name
code/article desync → `Открытых вопросов chunk-026 = 2` (OQ#1 SKU 15
из b2 + OQ#2 SKU 38+39 из b5) НЕ меняется, нового нумерованного OQ не
вводится.
