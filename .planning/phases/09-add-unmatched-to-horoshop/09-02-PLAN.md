<objective>
Layer SMART category resolution onto the Phase 9 core (plan 09-01), behind the SAME
`category_resolver` interface, so every unmatched card lands in the CORRECT `Раздел` instead of the
holding-category fallback. Three pluggable tiers in a chain `feed_category → analogy → fallback`,
plus an **AI (NVIDIA) resolver shipped DISABLED and gated**. Prove the analogy tier on REAL data and
hand Yana a written proposal (analogy vs AI vs hybrid) so she decides whether to enable AI.

Purpose: REQ-03 (every row a CORRECT category, not just a fallback) + REQ-06 (real-data evidence +
proposal; AI not default without Yana's go-ahead). This plan does NOT change the builder's output
schema — it only makes `resolver.resolve(sp)` smarter and wires the smart chain into `build_add_file`
via `build_resolver`.

Output: extended `np_parser`, a `category_export` reader, the smart resolvers in `category_resolver`,
an evidence audit script, tests, and `CATEGORY-PROPOSAL.md`. This plan is **NON-autonomous**: it ends
at a `checkpoint:decision` where Yana picks the strategy / whether to enable AI.

KEY FINDING (RESEARCH addendum): for NP — the ~205-item bulk of this phase — the feed itself carries
the category (`categories_uk`) and name (`title_uk`/`title_ru`), so feed_category resolves most NP
rows with NO analogy needed. Analogy carries non-NP suppliers. BUT the NP feed's category wording
differs from the store tree (feed «Холодильне обладнання/Льодогенератори» vs store «Холодильне та
морозильне обладнання/Льодогенератори») and Horoshop will NOT create missing categories — so
feed_category MUST reconcile to the store tree, never push verbatim.
</objective>
